# 20 — White-label Public Tracking

**Status:** Product specification  
**Public route:** `/t/{token}` UI backed by `GET /v1/track/{token}`  
**Authentication:** No account; possession of an unguessable token grants narrowly scoped read access.

## Scope and boundaries

In scope: recipient-facing status timeline, ETA, permitted rider/location context, merchant branding, localization, proof summary, support contact, accessibility, token lifecycle, caching controls, and custom-domain readiness.

Out of scope: recipient accounts, full delivery management, exposing operations/rider/route details, broad proof downloads, payment collection, marketing analytics, and treating a human order number or `external_order_id` as authorization.

## Actors

- Recipient opens the tracking link without signing in.
- Merchant admin configures approved branding and support details.
- Merchant support shares/regenerates a link under policy.
- Rider/operations updates the delivery that feeds tracking state.
- Platform admin moderates branding/custom domains and monitors abuse.
- Automated link scanners may access links and must not trigger mutations.

## Data and contracts

`TrackingToken` includes delivery ID, tenant/environment, random token hash, display-safe prefix, status, issued/expiry/revoked timestamps, generation version, and optional replacement reference. Store only a cryptographic hash; the raw token appears only in the URL.

Use at least 128 bits of cryptographically secure randomness, encoded URL-safely. Token lookup is constant-time where practical. Tokens are unrelated to delivery IDs, customer data, `external_order_id`, or order number.

`TrackingBrand` includes business name, approved logo asset, primary/accent colors, optional custom hostname, support contact, locale/timezone defaults, and accessibility-derived contrast settings.

Public payload is allowlisted:

- merchant display name/branding and masked order reference;
- normalized public status and safe timeline timestamps;
- ETA/window text with freshness/uncertainty;
- pickup/dropoff locality or masked address, never unnecessary full details;
- limited last-known location only in allowed lifecycle states;
- post-delivery proof summary where consent/policy permits;
- support contact and cache/version metadata.

No internal notes, COD ledger data, rider phone, other stops, route geometry, exact home coordinates, tenant IDs, or staff identities are returned.

## Endpoints and events

- `GET /v1/track/{token}` is unauthenticated and read-only.
- `GET /t/{token}` renders the web experience.
- `POST /v1/deliveries/{delivery_id}/tracking-token/rotate` requires merchant/admin authorization and idempotency.
- `GET/PATCH /v1/businesses/{business_id}/tracking-brand` manages branding.
- Custom-domain verification endpoints may be added in Phase 4.

Tracking consumes delivery lifecycle and ETA/location updates but does not emit merchant integration events for every page view. Internal events include `tracking_token.issued`, `tracking_token.rotated`, `tracking_token.revoked`, `tracking_brand.updated`, and `tracking_domain.verified`.

## Security

- Tokens are unguessable bearer capabilities, generated by a CSPRNG, hashed at rest, tenant/environment scoped, revocable, and rotatable.
- Enforce TLS, strict transport security, `Referrer-Policy: no-referrer`, restrictive CSP, frame protections, MIME protections, and no third-party scripts by default.
- Responses use `Cache-Control: private, no-store` for sensitive/live data. Service workers and analytics must not persist tokenized URLs.
- Rate-limit by privacy-preserving network signals/token prefix and detect enumeration. Invalid, expired, and revoked tokens return indistinguishable `404`.
- Scrub token paths/query/referrers from application, proxy, CDN, analytics, error, and support logs. Never place PII in page title, URL, or social preview.
- Logo assets are scanned/re-encoded; custom CSS/HTML/JavaScript is forbidden. Custom domains require DNS proof and managed TLS.

## Validation

- Token parser enforces exact alphabet/length/version before hash lookup.
- Public state is projected from the authoritative delivery timeline and approved visibility rules; unknown internal states map to safe public copy.
- ETA includes generated-at time and becomes “updating” when stale. Exact location is withheld before pickup/after completion and when policy disables it.
- Branding validates file type/dimensions/size, color contrast, business ownership, support links, locale, and hostname proof.
- Proof summary must pass retention, recipient-visibility, and content-safety rules before display.

## Error semantics

- Invalid, unknown, expired, revoked, and cross-environment tokens return the same generic `404 tracking_not_found`.
- Temporary projection failure returns generic `503 tracking_temporarily_unavailable` with request ID and retry guidance, without leaking delivery existence.
- Missing ETA/location is represented as unavailable/stale content, not a failed page.
- Unsupported branding falls back to safe platform defaults; one tenant's branding failure cannot break tracking for others.
- The public endpoint never returns stack traces, internal IDs, provider errors, or validation details useful for enumeration.

## Retry and idempotency

- `GET` is safe to retry. The page uses bounded polling with backoff/jitter, pauses when hidden, honors `Retry-After`, and stops/reduces after terminal state.
- Token rotation requires `Idempotency-Key`; repeated identical rotation returns the same replacement token only within the secure creation response contract, otherwise operators generate a new rotation intentionally.
- Projection/event consumers deduplicate event IDs and derive current state from authoritative delivery records to tolerate out-of-order updates.
- Frontend reconnects after transient failures without regressing the visible timeline or duplicating milestones.
- Link previews/scanners issue no mutations, notifications, or “viewed” business transitions.

## UI and admin touchpoints

- Public page is mobile-first and accessible: logo, masked order reference, status stepper, clear current status, scheduled window/ETA freshness, optional map, safe timeline, proof summary, and support details.
- The page supports merchant locale and timezone with platform fallback and avoids deceptive “live” wording for stale data.
- Delivery detail in `/app` provides copy/open/regenerate/revoke tracking link actions and records who rotated it.
- `/app/settings/branding` previews desktop/mobile states, validates contrast/assets, and controls permitted location/proof display.
- `/admin` handles domain verification, asset moderation, abuse/traffic anomalies, and emergency branding fallback without exposing raw tokens.

## Observability

- Metrics: page/API availability, latency, token valid/invalid ratio, rate limits, projection lag, stale ETA, location freshness, branding fallback, asset failures, and browser errors.
- Use short-lived request/session correlation IDs; do not log raw token, recipient network address beyond approved security retention, full user agent, or PII.
- Synthetic monitors use dedicated non-production fixtures and tokens. Alert on public SLO breach, enumeration spikes, token leakage indicators, stale projection, and custom-domain/TLS failure.
- Audit token issuance/rotation/revocation, branding changes, location/proof visibility changes, and domain verification.

## Phased delivery

1. **Foundation:** unguessable hashed token, no-login page/API, status stepper, masked references/addresses, platform branding, secure headers, and terminal polling behavior.
2. **Reliable tracking:** live ETA/location with freshness rules, proof summary, localization/timezones, accessibility testing, abuse detection, and public SLOs.
3. **White label:** approved logo/colors/support details, previews, asset pipeline, per-tenant visibility policy, and branded notifications.
4. **Scale:** verified custom domains, regional edge delivery with safe caching, richer localization, and privacy-preserving aggregate analytics.

## Acceptance criteria

- A recipient can open a link without an account and see only the allowlisted state for one delivery.
- Tokens have sufficient entropy, are stored hashed, cannot be derived from IDs/order data, and invalid variants return indistinguishable `404`.
- Raw tokens and recipient PII are absent from logs, analytics, referrers, caches, page titles, and third-party requests.
- Tracking never exposes another stop, route, tenant, rider contact, internal note, or precise location outside visibility policy.
- Polling survives transient errors, respects rate limits, and never regresses completed timeline states.
- Merchant branding remains accessible and safe; invalid assets/configuration fall back to platform defaults.
- Link rotation/revocation, custom-domain changes, and visibility settings are permissioned, idempotent where applicable, and audited.
